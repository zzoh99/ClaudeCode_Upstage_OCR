# 🧾 영수증 지출 관리 앱 (Receipt Expense Tracker)
### 프로그램 개요서 | Ver 1.0 | 2025

---

## 1. 프로젝트 개요

### 1.1 프로그램 목적

영수증(이미지/PDF)을 업로드하면 **Upstage Vision LLM**이 자동으로 내용을 인식하여 구조화된 지출 데이터로 변환하고,
사용자가 지출 내역을 손쉽게 조회·관리할 수 있는 **경량 웹 애플리케이션**입니다.

### 1.2 핵심 가치

| 가치 | 설명 |
|------|------|
| **자동화** | 영수증을 수동으로 입력하지 않고 AI가 자동 파싱 |
| **단순함** | DB 없이 JSON 파일 기반으로 경량 운영 |
| **빠른 배포** | Vercel + GitHub 연동으로 즉시 서비스 가능 |
| **확장성** | FastAPI + LangChain 기반으로 기능 추가 용이 |

### 1.3 타겟 사용자

- 개인 지출 내역을 간편하게 관리하고 싶은 일반 사용자
- 복잡한 가계부 앱 없이 영수증만으로 소비 패턴을 파악하고 싶은 사용자
- AI 기반 OCR 기술을 활용한 사이드 프로젝트를 원하는 개발자

### 1.4 개발 조건 요약

| 항목 | 내용 |
|------|------|
| 개발 기간 | 1일 (단기 스프린트) |
| 데이터베이스 | 미사용 (JSON 파일 기반 저장) |
| 배포 환경 | Vercel (프론트 + 백엔드 서버리스) |
| 협업 도구 | GitHub Repository |

## 4. 시스템 아키텍처

### 4.1 전체 흐름

```
┌──────────────────────────────────────────────────────┐
│                   사용자 브라우저                       │
│        React + Vite + TailwindCSS + Axios            │
└────────────────────┬─────────────────────────────────┘
                     │  HTTP (REST API)
                     ▼
┌──────────────────────────────────────────────────────┐
│              FastAPI 백엔드 서버                       │
│                                                      │
│  POST /api/upload  ──►  LangChain Chain              │
│                              │                       │
│                              ▼                       │
│                    Upstage Vision LLM                │
│                    (OCR + 구조화 추출)                 │
│                              │                       │
│                              ▼                       │
│                  JSON 파일 저장 (expenses.json)        │
│                                                      │
│  GET  /api/expenses ──► JSON 파일 읽기 → 응답          │
└──────────────────────────────────────────────────────┘
                     │  GitHub Push → 자동 배포
                     ▼
┌──────────────────────────────────────────────────────┐
│         GitHub Repository ──► Vercel 배포             │
└──────────────────────────────────────────────────────┘
```

### 4.2 LangChain 처리 흐름

```
영수증 파일 업로드
      │
      ▼
[이미지 전처리]  PIL / pdf2image → Base64 인코딩
      │
      ▼
[LangChain Chain]
      │
      ├─► ChatUpstage (Vision LLM) 호출
      │        System Prompt: "JSON 형식으로만 응답"
      │        User Input: Base64 이미지 + 추출 지시문
      │
      ▼
[구조화 JSON 파싱]  LangChain Output Parser
      │
      ▼
expenses.json 파일에 append 저장
```

## 5. API 설계 (엔드포인트)

| 메서드 | URL | 설명 | 요청 | 응답 |
|--------|-----|------|------|------|
| `POST` | `/api/upload` | 영수증 업로드 및 OCR 파싱 | `multipart/form-data (file)` | 파싱된 JSON 객체 |
| `GET` | `/api/expenses` | 전체 지출 내역 조회 | 쿼리: `?from=&to=` | 지출 목록 배열 |
| `DELETE` | `/api/expenses/{id}` | 특정 지출 내역 삭제 | Path param: `id` | 삭제 성공 메시지 |
| `PUT` | `/api/expenses/{id}` | 지출 내역 수정 | JSON body | 수정된 객체 |
| `GET` | `/api/summary` | 지출 합계 통계 조회 | 쿼리: `?month=` | 합계, 카테고리별 통계 |

### 6.2 데이터 저장 방식 (DB 미사용)

> ⚠️ **DB 미사용 데이터 처리 전략**
>
> - **서버 측**: `expenses.json` 파일에 배열 형태로 누적 저장 (append 방식)
> - **클라이언트 측**: React `useState`로 실시간 UI 반영
> - **MVP 해결책 A**: 클라이언트 `localStorage`에 데이터를 병행 저장하여 프론트에서 영속성 유지
> - **MVP 해결책 B**: 백엔드를 Railway, Render 등 일반 서버에 배포하여 파일 시스템 유지
> - **장기 확장**: Vercel KV(Redis) 또는 Supabase(무료 플랜) 도입 시 DB 전환 권장

## 10. 제약사항 및 가정

### 10.1 DB 미사용에 따른 제약

> **데이터 영속성 주의사항**
>
> Vercel 서버리스 함수는 요청마다 새로운 컨테이너에서 실행되어 파일 시스템이 유지되지 않을 수 있습니다.
> 아래 대응 방안 중 하나를 선택하세요.
>
> | 방안 | 설명 | 난이도 |
> |------|------|--------|
> | localStorage 병행 저장 | 클라이언트에서 영속성 유지 | ⭐ 쉬움 |
> | Railway / Render 배포 | 일반 서버에서 파일 시스템 유지 | ⭐⭐ 보통 |
> | Vercel KV (Redis) | Vercel 내장 키-값 저장소 활용 | ⭐⭐ 보통 |
> | Supabase 무료 플랜 | PostgreSQL DB로 영구 전환 | ⭐⭐⭐ 어려움 |

### 10.2 기타 가정 및 제약

- Upstage API Key는 개인 발급하여 Vercel 환경변수에 등록한다고 가정
- 영수증 이미지는 JPG, PNG, PDF만 지원 (최대 10MB)
- 다국어 영수증 지원은 1차 범위에서 제외 (한국어/영어 영수증만 지원)
- 동시 접속 사용자 수는 1인 기준 (개인 프로젝트)
- 보안 인증(로그인)은 1일 범위에서 제외하며, API URL 비공개로 최소 보안 유지
